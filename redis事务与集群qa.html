Redis 事务与集群 QA

Q1: Redis的事务机制是怎样的？有什么特点和局限性？

A1:

Redis事务特点

Redis事务是一组命令的集合，具有以下特点：

1. ACID特性分析

  • 原子性（Atomicity）: 事务中的命令要么全部执行，要么全部不执行

  • 一致性（Consistency）: 事务执行前后数据保持一致

  • 隔离性（Isolation）: 事务之间相互隔离，不会相互影响

  • 持久性（Durability）: 事务执行结果会被持久化（取决于持久化配置）

2. 事务命令

  • MULTI: 开启事务，后续命令进入队列

  • EXEC: 执行事务队列中的所有命令

  • DISCARD: 取消事务，清空命令队列

  • WATCH: 监视key，实现乐观锁机制

事务执行流程

事务示例

# 正常事务执行 127.0.0.1:6379> MULTI OK 127.0.0.1:6379> SET account:1001 100 QUEUED 127.0.0.1:6379> SET account:1002 200 QUEUED 127.0.0.1:6379> EXEC 1) OK 2) OK # 事务取消 127.0.0.1:6379> MULTI OK 127.0.0.1:6379> SET key1 value1 QUEUED 127.0.0.1:6379> DISCARD OK

事务的局限性

1. 不支持回滚

# 即使某个命令执行失败，其他命令仍会执行 127.0.0.1:6379> MULTI OK 127.0.0.1:6379> SET key1 value1 QUEUED 127.0.0.1:6379> INCR key1 # 这会失败，因为key1是字符串 QUEUED 127.0.0.1:6379> SET key2 value2 QUEUED 127.0.0.1:6379> EXEC 1) OK 2) (error) ERR value is not an integer or out of range 3) OK # 这个命令仍然执行成功

2. 命令错误处理

  • 语法错误: 整个事务都不会执行

  • 运行时错误: 只有错误的命令失败，其他命令正常执行

3. 无法在事务中使用条件判断

Q2: WATCH命令如何实现乐观锁？在什么场景下使用？

A2:

WATCH命令原理

WATCH命令实现了乐观锁机制，用于解决并发修改问题：

1. 工作原理

  • 监视一个或多个key

  • 如果在事务执行前，被监视的key发生了变化，事务会被取消

  • 基于版本号机制，每次修改key都会更新版本

2. 使用流程

WATCH key1 key2 # 监视key MULTI # 开启事务 # ... 事务命令 EXEC # 执行事务（可能失败）

乐观锁实现示例

场景：银行转账

import redis def transfer_money(from_account, to_account, amount): r = redis.Redis() while True: # 监视账户余额 r.watch(from_account, to_account) # 获取当前余额 from_balance = int(r.get(from_account) or 0) to_balance = int(r.get(to_account) or 0) # 检查余额是否足够 if from_balance < amount: r.unwatch() # 取消监视 return False, "余额不足" # 开启事务 pipe = r.pipeline() pipe.multi() # 执行转账 pipe.set(from_account, from_balance - amount) pipe.set(to_account, to_balance + amount) try: # 执行事务 result = pipe.execute() return True, "转账成功" except redis.WatchError: # 如果key被修改，重试 continue

WATCH的执行流程

应用场景

1. 库存扣减

def decrease_stock(product_id, quantity): r = redis.Redis() stock_key = f"stock:{product_id}" while True: r.watch(stock_key) current_stock = int(r.get(stock_key) or 0) if current_stock < quantity: r.unwatch() return False, "库存不足" pipe = r.pipeline() pipe.multi() pipe.set(stock_key, current_stock - quantity) try: pipe.execute() return True, "扣减成功" except redis.WatchError: continue

2. 计数器更新

def increment_counter_with_limit(counter_key, max_value): r = redis.Redis() while True: r.watch(counter_key) current_value = int(r.get(counter_key) or 0) if current_value >= max_value: r.unwatch() return False, "已达到最大值" pipe = r.pipeline() pipe.multi() pipe.set(counter_key, current_value + 1) try: pipe.execute() return True, current_value + 1 except redis.WatchError: continue

WATCH的注意事项

1. 性能考虑

  • 高并发场景下可能导致大量重试

  • 适合冲突较少的场景

  • 可以设置最大重试次数

2. 内存影响

  • 每个被WATCH的key都会消耗额外内存

  • 及时使用UNWATCH释放监视

Q3: Redis主从复制的原理是什么？如何配置和监控？

A3:

主从复制原理

Redis主从复制是实现数据备份和读写分离的基础机制：

1. 复制过程

全量同步（Full Resynchronization）

  • 从节点首次连接主节点时触发

  • 主节点生成RDB快照发送给从节点

  • 从节点加载RDB文件恢复数据

增量同步（Partial Resynchronization）

  • 正常运行时的数据同步

  • 主节点将写命令发送给从节点

  • 从节点执行命令保持数据一致

2. 复制流程详解

主从配置

1. 从节点配置

# 方式1：配置文件 replicaof 192.168.1.100 6379 # 方式2：命令行 redis-server --replicaof 192.168.1.100 6379 # 方式3：运行时配置 127.0.0.1:6379> REPLICAOF 192.168.1.100 6379

2. 主节点配置

# 设置复制积压缓冲区大小（默认1MB） repl-backlog-size 1mb # 设置复制积压缓冲区TTL（默认3600秒） repl-backlog-ttl 3600 # 设置从节点优先级（用于故障转移） replica-priority 100 # 配置密码认证 requirepass mypassword masterauth mypassword

3. 安全配置

# 主节点密码 requirepass master_password # 从节点连接主节点的密码 masterauth master_password # 只读模式（从节点默认开启） replica-read-only yes

监控主从复制

1. 主节点监控

# 查看复制信息 redis-cli INFO replication # 关键指标 connected_slaves: 连接的从节点数量 master_repl_offset: 主节点复制偏移量 repl_backlog_active: 复制积压缓冲区是否激活 repl_backlog_size: 复制积压缓冲区大小

2. 从节点监控

# 查看复制状态 redis-cli INFO replication # 关键指标 master_link_status: 与主节点连接状态（up/down） master_last_io_seconds_ago: 最后一次与主节点通信时间 master_sync_in_progress: 是否正在同步 slave_repl_offset: 从节点复制偏移量 slave_lag: 复制延迟

3. 监控脚本示例

import redis import time def monitor_replication(): master = redis.Redis(host='192.168.1.100', port=6379) slave = redis.Redis(host='192.168.1.101', port=6379) while True: try: # 获取主从复制信息 master_info = master.info('replication') slave_info = slave.info('replication') # 检查连接状态 if slave_info['master_link_status'] != 'up': print("警告：从节点与主节点连接断开") # 检查复制延迟 master_offset = master_info['master_repl_offset'] slave_offset = slave_info['slave_repl_offset'] lag = master_offset - slave_offset if lag > 1000: # 延迟超过1000字节 print(f"警告：复制延迟过大，延迟：{lag}字节") # 检查最后通信时间 last_io = slave_info['master_last_io_seconds_ago'] if last_io > 30: # 超过30秒未通信 print(f"警告：主从通信异常，{last_io}秒未通信") print(f"主节点偏移量：{master_offset}") print(f"从节点偏移量：{slave_offset}") print(f"复制延迟：{lag}字节") print("---") except Exception as e: print(f"监控异常：{e}") time.sleep(10)

Q4: Redis哨兵模式是如何工作的？如何实现自动故障转移？

A4:

哨兵模式概述

Redis Sentinel是Redis的高可用解决方案，提供以下功能：

1. 核心功能

  • 监控（Monitoring）: 监控主从节点状态

  • 通知（Notification）: 故障时发送通知

  • 自动故障转移（Automatic Failover）: 主节点故障时自动选举新主节点

  • 配置提供（Configuration Provider）: 为客户端提供当前主节点信息

2. 哨兵集群架构

故障转移流程

1. 故障检测

2. 领导者选举

3. 故障转移执行

哨兵配置

1. 哨兵配置文件（sentinel.conf）

# 哨兵端口 port 26379 # 监控主节点 sentinel monitor mymaster 192.168.1.100 6379 2 # 主节点密码 sentinel auth-pass mymaster mypassword # 主观下线时间（毫秒） sentinel down-after-milliseconds mymaster 30000 # 故障转移超时时间（毫秒） sentinel failover-timeout mymaster 180000 # 并行同步的从节点数量 sentinel parallel-syncs mymaster 1 # 通知脚本 sentinel notification-script mymaster /var/redis/notify.sh # 故障转移脚本 sentinel client-reconfig-script mymaster /var/redis/reconfig.sh

2. 启动哨兵

# 启动哨兵 redis-sentinel /path/to/sentinel.conf # 或者 redis-server /path/to/sentinel.conf --sentinel

3. 客户端连接配置

import redis.sentinel # 哨兵列表 sentinels = [ ('192.168.1.101', 26379), ('192.168.1.102', 26379), ('192.168.1.103', 26379) ] # 创建哨兵连接 sentinel = redis.sentinel.Sentinel(sentinels) # 获取主节点连接 master = sentinel.master_for('mymaster', socket_timeout=0.1) # 获取从节点连接 slave = sentinel.slave_for('mymaster', socket_timeout=0.1) # 使用连接 master.set('key', 'value') value = slave.get('key')

哨兵监控和运维

1. 哨兵状态查询

# 连接哨兵 redis-cli -p 26379 # 查看监控的主节点 SENTINEL masters # 查看从节点信息 SENTINEL slaves mymaster # 查看其他哨兵信息 SENTINEL sentinels mymaster # 手动故障转移 SENTINEL failover mymaster

2. 监控脚本

import redis.sentinel def monitor_sentinel(): sentinels = [('localhost', 26379)] sentinel = redis.sentinel.Sentinel(sentinels) try: # 获取主节点信息 master_info = sentinel.sentinel_masters() for master_name, info in master_info.items(): print(f"主节点 {master_name}:") print(f" 地址: {info['ip']}:{info['port']}") print(f" 状态: {info['flags']}") print(f" 最后检测: {info['last-ping-reply']}ms前") # 获取从节点信息 slaves = sentinel.sentinel_slaves('mymaster') print(f"从节点数量: {len(slaves)}") except Exception as e: print(f"监控异常: {e}")
