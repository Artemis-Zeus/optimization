Q2: 如何配置MySQL主从复制？包括具体的配置步骤？

答案：

主从复制配置步骤：

具体配置过程：

  1. 主库配置（Master）

    # my.cnf配置 [mysqld] server-id = 1 # 唯一标识 log-bin = mysql-bin # 开启二进制日志 binlog-format = ROW # 推荐ROW格式 expire_logs_days = 7 # 日志保留天数 max_binlog_size = 100M # 单个日志文件大小
    -- 创建复制用户 CREATE USER 'repl'@'%' IDENTIFIED BY 'replication_password'; GRANT REPLICATION SLAVE ON *.* TO 'repl'@'%'; FLUSH PRIVILEGES; -- 锁定主库并获取binlog位置 FLUSH TABLES WITH READ LOCK; SHOW MASTER STATUS; -- 记录File和Position值 UNLOCK TABLES;

  2. 从库配置（Slave）

    # my.cnf配置 [mysqld] server-id = 2 # 不同于主库的ID relay-log = relay-bin # 中继日志 read-only = 1 # 只读模式
    -- 配置主库连接信息 CHANGE MASTER TO MASTER_HOST='192.168.1.100', MASTER_USER='repl', MASTER_PASSWORD='replication_password', MASTER_LOG_FILE='mysql-bin.000001', MASTER_LOG_POS=154; -- 启动复制 START SLAVE; -- 检查复制状态 SHOW SLAVE STATUS\G

Q3: 主从复制延迟问题如何解决？有哪些监控指标？

答案：

复制延迟产生原因：

监控指标和命令：

-- 查看复制状态 SHOW SLAVE STATUS\G -- 重要监控指标 SELECT CASE WHEN Slave_IO_Running = 'Yes' AND Slave_SQL_Running = 'Yes' THEN 'OK' ELSE 'ERROR' END as Replication_Status, Seconds_Behind_Master as Lag_Seconds, Master_Log_File, Read_Master_Log_Pos, Relay_Master_Log_File, Exec_Master_Log_Pos FROM (SHOW SLAVE STATUS) as slave_status; -- 计算复制延迟 SELECT TIMESTAMPDIFF(SECOND, STR_TO_DATE(SUBSTRING_INDEX(SUBSTRING_INDEX(@@hostname, '-', -1), '.', 1), '%Y%m%d'), NOW()) as estimated_lag; -- 监控复制线程状态 SHOW PROCESSLIST;

并行复制优化：

-- MySQL 5.7+ 并行复制配置 SET GLOBAL slave_parallel_type = 'LOGICAL_CLOCK'; SET GLOBAL slave_parallel_workers = 8; SET GLOBAL slave_preserve_commit_order = ON; -- 查看并行复制状态 SELECT * FROM performance_schema.replication_applier_status_by_worker;

MHA配置步骤：

  1. 安装MHA软件

    # 在所有节点安装MHA Node yum install mha4mysql-node # 在管理节点安装MHA Manager yum install mha4mysql-manager

  2. 配置SSH免密登录

    # 在所有节点间配置SSH免密登录 ssh-keygen -t rsa ssh-copy-id root@mysql-node1 ssh-copy-id root@mysql-node2 ssh-copy-id root@mysql-node3

  3. MHA配置文件

    # /etc/mha/app1.cnf [server default] manager_log=/var/log/mha/app1/manager.log manager_workdir=/var/log/mha/app1 master_binlog_dir=/var/lib/mysql password=mha_password user=mha ping_interval=3 repl_password=replication_password repl_user=repl ssh_user=root [server1] hostname=192.168.1.101 port=3306 candidate_master=1 [server2] hostname=192.168.1.102 port=3306 candidate_master=1 [server3] hostname=192.168.1.103 port=3306 no_master=1

  4. 启动MHA监控

    # 检查配置 masterha_check_ssh --conf=/etc/mha/app1.cnf masterha_check_repl --conf=/etc/mha/app1.cnf # 启动MHA管理器 masterha_manager --conf=/etc/mha/app1.cnf --daemon # 查看状态 masterha_check_status --conf=/etc/mha/app1.cnf

-- 查看集群状态 SHOW STATUS LIKE 'wsrep%'; -- 重要状态指标 SELECT VARIABLE_NAME, VARIABLE_VALUE FROM INFORMATION_SCHEMA.GLOBAL_STATUS WHERE VARIABLE_NAME IN ( 'wsrep_cluster_size', 'wsrep_cluster_status', 'wsrep_connected', 'wsrep_ready', 'wsrep_local_state_comment' ); -- 节点加入和离开 -- 新节点加入集群会自动同步数据 -- 使用SST (State Snapshot Transfer) 或 IST (Incremental State Transfer)

Q10: 如何制定完整的MySQL备份和恢复策略？

答案：

备份策略设计：

完整备份策略示例：

#!/bin/bash # 备份脚本示例 # 1. 全量备份（每周日执行） if [ $(date +%w) -eq 0 ]; then mysqldump --single-transaction --routines --triggers \ --all-databases > /backup/full_backup_$(date +%Y%m%d).sql # 使用XtraBackup物理备份 innobackupex --user=backup --password=backup_pass /backup/xtrabackup/ fi # 2. 增量备份（每日执行） if [ $(date +%w) -ne 0 ]; then # 备份binlog mysqlbinlog --start-datetime="$(date -d yesterday +%Y-%m-%d) 00:00:00" \ --stop-datetime="$(date +%Y-%m-%d) 00:00:00" \ /var/lib/mysql/mysql-bin.* > /backup/incremental_$(date +%Y%m%d).sql fi # 3. 实时binlog备份 mysqlbinlog --read-from-remote-server --host=localhost \ --stop-never mysql-bin.000001 > /backup/realtime_binlog.sql & # 4. 自动清理老备份 find /backup -name "*.sql" -mtime +30 -delete

恢复策略和测试：

-- 1. 全量恢复 mysql < /backup/full_backup_20231201.sql -- 2. 点时间恢复 mysql < /backup/full_backup_20231201.sql mysqlbinlog --start-datetime="2023-12-01 10:00:00" \ --stop-datetime="2023-12-01 15:30:00" \ /backup/mysql-bin.000123 | mysql -- 3. 表级恢复 mysqldump --single-transaction db_name table_name | mysql db_name -- 4. 验证恢复结果 SELECT COUNT(*) FROM critical_table; SELECT MAX(created_at) FROM transaction_log;

备份监控和告警：

# 备份监控脚本 #!/bin/bash # 检查备份文件是否存在 backup_file="/backup/full_backup_$(date +%Y%m%d).sql" if [ ! -f "$backup_file" ]; then echo "ERROR: Backup file not found!" | mail -s "Backup Alert" admin@company.com fi # 检查备份文件大小 file_size=$(stat -f%z "$backup_file" 2>/dev/null || stat -c%s "$backup_file") if [ $file_size -lt 1000000 ]; then # 小于1MB认为异常 echo "WARNING: Backup file too small!" | mail -s "Backup Alert" admin@company.com fi # 测试备份文件完整性 mysql --execute="source $backup_file" test_db_restore if [ $? -ne 0 ]; then echo "ERROR: Backup file corrupted!" | mail -s "Backup Alert" admin@company.com fi
